# Phase 12.2 Research

## Problem

The Scoop backend is now much cleaner than before, but two files still carry too much mixed responsibility:

- `spoon-backend/src/scoop/host/surface.rs`
- `spoon-backend/src/scoop/actions.rs`
- `spoon-backend/src/scoop/extract.rs`
- `spoon-backend/src/scoop/info.rs`
- `spoon-backend/src/scoop/buckets.rs`

At the same time, parts of `scoop/lifecycle/` may now be so thin that they no longer justify their own files.

## Findings

### `host/surface.rs`
Current contents span:

- shim target resolution
- shim writing/removal
- shortcut writing/removal
- environment substitution helpers
- command-surface reapply orchestration

This is too much for one file, even if the domain umbrella is still “surface”.

### `actions.rs`
Current contents span:

- operation entry
- dependency walk
- install/update orchestration
- uninstall orchestration
- journal/lock usage
- lifecycle stage emission

This is still the right domain root, but too much for one file.

### `extract.rs`
Current contents span:

- archive kind detection
- 7z/MSI helper extraction
- install-root materialization
- current-entry refresh

This is a second high-value decomposition target because it mixes archive mechanics with install-root orchestration.

### `info.rs`
Current contents span:

- package operation outcome shaping
- package details aggregation
- manifest/display helpers
- installed/runtime detail reconciliation

This file is large enough that human navigation cost is now significant.

### `buckets.rs`
Current contents span:

- bucket registry persistence
- bucket add/remove/update orchestration
- bucket sync behavior
- manifest resolution

This is another file whose current size and mixed concerns now justify decomposition.

### `lifecycle/`
Some files still earn their keep through phase semantics:

- acquire
- materialize
- persist
- surface

But others are close to adapter shells:

- reapply
- uninstall

## Recommended Direction

1. Split `host/surface.rs` into focused modules.
2. Convert `actions.rs` into an `actions/` directory with a thin root module.
3. Split `extract.rs` into focused modules if the boundaries stay clear.
4. Split `info.rs` into cleaner detail/outcome/read-model support slices.
5. Split `buckets.rs` into cleaner registry/store, sync/update, and resolution slices.
6. Collapse only lifecycle files that no longer contribute meaningful stage semantics.

## Success Signal

After this phase, a reader should be able to answer:

- where shims are handled
- where shortcuts are handled
- where action orchestration begins
- where archive/materialization logic begins
- where package detail/read-model shaping begins
- where bucket registry and sync logic begin
- which lifecycle files still represent real phases

without opening a giant file first.
