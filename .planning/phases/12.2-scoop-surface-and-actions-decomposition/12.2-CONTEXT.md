# Phase 12.2: Scoop Surface and Actions Decomposition - Context

**Gathered:** 2026-04-04
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase continues the Scoop cleanup by decomposing the remaining oversized orchestration files without reopening another full architecture redesign.

The goal is to improve human readability and boundary clarity in the Scoop backend by:

- splitting the oversized `host/surface.rs`
- decomposing `actions.rs` into clearer submodules
- decomposing the oversized `extract.rs` helper cluster
- decomposing the oversized `info.rs` detail aggregation layer
- decomposing the oversized `buckets.rs` bucket inventory / registry / sync layer
- collapsing any remaining lifecycle adapter shells that no longer carry real stage value

This phase is about **structure and readability**, not new features.

</domain>

<decisions>
## Implementation Decisions

### Surface Decomposition
- **D-01:** `scoop/host/surface.rs` is too large and mixes multiple responsibilities.
- **D-02:** The preferred direction is to split it into focused modules rather than keep one large file.
- **D-03:** The most likely split is:
  - shim target expansion / shim writing
  - shortcut creation / removal
  - command-surface reapply orchestration

### Extract Decomposition
- **D-04:** `scoop/extract.rs` is also large enough to justify decomposition.
- **D-05:** The preferred direction is to separate:
  - archive extraction helpers
  - install-root materialization
  - current-root refresh / entry maintenance
- **D-06:** The decomposition should preserve the existing shared/archive boundary rather than re-introduce domain confusion.

### Actions Decomposition
- **D-07:** `scoop/actions.rs` should remain the domain-root operation coordinator.
- **D-08:** This phase should not move the entire action story back into `lifecycle/`.
- **D-09:** Instead, `actions.rs` should become an `actions/` module family with the root module preserving the coordinator role.

### Lifecycle Thin Layers
- **D-10:** Thin lifecycle modules that still provide real stage meaning may remain.
- **D-11:** Lifecycle files that are now only adapter shells should be collapsed into `actions/` or other clearer homes.
- **D-12:** The likely first candidates for collapse are `reapply` and `uninstall`.

### Readability Guardrail
- **D-13:** Splitting files is only worthwhile if the resulting structure is easier for a human to navigate.
- **D-14:** Avoid replacing one large file with many meaningless micro-files.

### Error / Naming Follow-through
- **D-15:** Continue honoring the newer naming direction:
  - `ResolvedPackageSource`
  - `PackageAsset`
  - `Db`
- **D-16:** If a split exposes additional stale `payload` or weak error naming, fix it in the same phase.

### Additional Large Files
- **D-17:** `info.rs` is in-scope for real decomposition during this phase, not just passive evaluation.
- **D-18:** `buckets.rs` is also in-scope for decomposition during this phase.
- **D-19:** `surface/actions/extract/info/buckets` form the complete decomposition target set for 12.2.
- **D-20:** Execution order can still prioritize `surface/actions` first, but `info/buckets` should not be deferred out of the phase.

</decisions>

<code_context>
## Existing Code Insights

### Highest-pressure files
- `spoon-backend/src/scoop/host/surface.rs`
- `spoon-backend/src/scoop/actions.rs`
- `spoon-backend/src/scoop/extract.rs`
- `spoon-backend/src/scoop/info.rs`
- `spoon-backend/src/scoop/buckets.rs`

### Nearby modules likely to be affected
- `spoon-backend/src/scoop/lifecycle/surface.rs`
- `spoon-backend/src/scoop/lifecycle/reapply.rs`
- `spoon-backend/src/scoop/lifecycle/uninstall.rs`
- `spoon-backend/src/scoop/lifecycle/acquire.rs`
- `spoon-backend/src/scoop/lifecycle/materialize.rs`
- `spoon-backend/src/scoop/host/mod.rs`
- `spoon-backend/src/scoop/mod.rs`
- `spoon/src/service/scoop/runtime.rs`

### Current shape
- `surface.rs` mixes shim discovery, shim writing, shortcut handling, and command-surface reapply orchestration.
- `actions.rs` is the right root-level coordinator, but it now contains enough orchestration that internal directory decomposition would improve readability.
- `extract.rs` mixes archive-type handling, helper-based extraction, install-root materialization, and current-entry refresh.
- `info.rs` is a large detail/read-model aggregator with many distinct sub-concerns living in one file.
- `buckets.rs` mixes registry persistence, bucket sync/update flows, and manifest resolution in one file.
- Some lifecycle modules still look like thin wrappers around lower or higher layers.

</code_context>

<specifics>
## Specific Ideas

- Prefer a `host/surface/` directory or a similarly explicit file split over keeping a single catch-all `surface.rs`.
- Prefer an `actions/` directory with a small `mod.rs` over one large `actions.rs`.
- Prefer an `extract/` split if it can separate archive helpers, materialization, and current-entry maintenance cleanly.
- Prefer an `info/` split if package operation outcome, package details, and manifest/detail shaping can be separated cleanly.
- Prefer a `buckets/` split if registry/store, sync/update orchestration, and manifest resolution can be separated cleanly.
- Keep `lifecycle` only where the file still buys genuine stage semantics.

</specifics>
