## SQLite Ownership Scope

| Option | Description | Selected |
|--------|-------------|----------|
| Full control-plane SQLite | SQLite owns installed state, journal/checkpoints, doctor/repair metadata, bucket registry/sync metadata, lock state, and schema metadata, while filesystem remains runtime data plane | x |
| Partial SQLite | Only installed state plus a subset of metadata move into SQLite | |
| Journal-only SQLite | SQLite stores journal/recovery, but installed state stays in JSON | |

**User's choice:** Full control-plane SQLite.
**Notes:** The user explicitly chose the most complete control-plane scope, while keeping install roots, `current`, persist, cache, shims, and bucket repos on the filesystem.

---

## Sync Core / Async Edge

| Option | Description | Selected |
|--------|-------------|----------|
| Lifecycle-first backend with sync core and async edges | Pure or near-pure planning/rules/projections stay sync-oriented; DB/filesystem/process/network stay at async edges | x |
| Async-first backend | Let async shape most of the core APIs and flow structure | |
| Mostly sync backend | Push async concerns to the very outer shell only | |

**User's choice:** Lifecycle-first backend with sync core and async edges.
**Notes:** The user agreed that sync core mainly means pure calculation and rule layers, not IO.

---

## SQLite Driver Choice

| Option | Description | Selected |
|--------|-------------|----------|
| `rusqlite` + owned tokio wrapper | Keep SQLite/rusqlite semantics and encapsulate blocking/async bridging inside backend store boundaries | x |
| `tokio-rusqlite` | Use a bridge crate to expose async SQLite calls on top of rusqlite | |
| `sqlx + sqlite` | Use async SQLx style as the main access layer | |

**User's choice:** `rusqlite` + owned tokio wrapper.
**Notes:** The user changed direction after comparing maintenance risk and chose to own the tokio boundary inside the repo instead of depending on `tokio-rusqlite`.

---

## Cutover Policy

| Option | Description | Selected |
|--------|-------------|----------|
| Direct cutover, no compatibility | Switch directly to SQLite control-plane state and require explicit operator repair/manual cleanup if old JSON state cannot be reconciled | x |
| Guided import/repair path | Provide a formal import tool or import-first workflow before full cutover | |
| Dual-read/dual-write compatibility | Keep old and new state stores alive during a transition period | |

**User's choice:** Direct cutover, no compatibility.
**Notes:** The user explicitly said not to consider compatibility. Manual deletion of old state is acceptable if necessary.
