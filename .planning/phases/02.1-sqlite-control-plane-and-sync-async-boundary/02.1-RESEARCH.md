# Phase 02.1: SQLite Control Plane and Sync-Async Boundary - Research

**Researched:** 2026-03-29
**Domain:** SQLite control-plane insertion for `spoon-backend`
**Confidence:** HIGH

## Summary

Phase 02.1 is best treated as a control-plane pivot, not a generic database migration. Phase 2 already established one canonical installed-state model, which means the project now has a clean metadata shape to move into SQLite without first untangling multiple live state contracts. That makes this the right moment to introduce SQLite before Phase 3 lifecycle splitting deepens the number of places that would otherwise keep talking to JSON files.

The storage split is explicit: SQLite should become the control plane, while the filesystem remains the data plane. Installed-state rows, journal/checkpoints, doctor/repair metadata, bucket registry metadata, lock state, and schema metadata belong in SQLite. Install roots, `current`, persist, shims, shortcuts, caches, and bucket repos remain on disk and continue to derive from `RuntimeLayout`. This is the cleanest split because it aligns with what SQLite is good at: authoritative metadata, transactions, checkpoints, and queryable repair state.

For the Rust integration, `rusqlite` is the right core driver, but the tokio async boundary should be owned by this repo rather than delegated to a bridge crate. The key architectural point is boundary choice: `rusqlite` should be contained inside store/repository facades that expose async methods while hiding `spawn_blocking` or equivalent worker-thread details. Sync-core modules such as state models, lifecycle planning, transition rules, ordering, and projections should stay driver-free. The async edge should own DB calls, filesystem actions, process execution, networking, and event emission.

The direct-cutover decision simplifies planning. There should be no dual-write and no dual-read fallback to JSON control files. Instead, 02.1 should introduce SQLite bootstrap and migrations, move installed state and bucket registry metadata into SQLite, add journal and doctor metadata stores, then switch active read/write paths to the DB. Leftover JSON control files should become explicit operator-visible stale-state issues that can be repaired or deleted manually. This matches the project's forward-design bias and prevents a long-lived split-brain control plane.

## Phase Requirements

| ID | Description | Planning Implication |
|----|-------------|----------------------|
| SQLCP-01 | SQLite becomes the backend control plane for installed state and recovery metadata. | Add backend-wide DB bootstrap plus domain stores. |
| SQLCP-02 | Filesystem remains runtime data plane. | Keep layout-derived roots and all runtime artifacts on disk. |
| SQLCP-03 | `rusqlite` stays encapsulated behind store/repository edges with repo-owned tokio bridging. | No driver types or blocking-bridge details in lifecycle rule modules. |
| SQLCP-04 | Preserve sync-core / async-edge boundaries. | Plans must explicitly separate models/projections from IO/store code. |
| SQLCP-05 | Direct cutover from JSON with no compatibility layer. | Add explicit stale JSON reporting and manual cleanup path. |

## Recommended Plan Order

1. Add control-plane bootstrap, migrations, and DB facade.
2. Move installed state and bucket registry metadata into SQLite.
3. Add operation journal, lock state, and doctor/repair metadata stores.
4. Cut read/write control-plane paths directly to SQLite and surface old JSON as repair-needed.

## Validation Focus

- Installed-state roundtrip through SQLite
- Operation journal row creation
- Bucket registry metadata roundtrip without moving repo directories
- Driver-free sync-core modules
- Legacy JSON control-plane files explicitly reported after cutover
