## Decisions Made
- **Error-type bridging:** `ControlPlaneDb::call()` accepts closures returning `rusqlite::Result<T>` and internally maps `rusqlite::Error` to `tokio_rusqlite::Error`, then to `crate::BackendError`. This hides both `tokio_rusqlite::Error` and `rusqlite::Error` from business-rule callers.
- **Bundled SQLite:** Using `rusqlite` with `bundled` feature for Windows portability -- no system libsqlite3 installation required.
- **WAL journal mode:** Enabled by default at connection open for better concurrent read performance.

**3. [Rule 3 - Blocking] Resolved tokio_rusqlite::Error vs rusqlite::Error type mismatch**
- **Found during:** Task 1 (multiple compilation attempts)
- **Issue:** `tokio_rusqlite::Error` is a separate enum (ConnectionClosed, Close, Rusqlite, Other) from `rusqlite::Error`. The `call()` bound expects closures returning `tokio_rusqlite::Result<T>`, but `rusqlite::Connection::execute_batch` returns `rusqlite::Result<()>`. Adding `rusqlite` as a direct dependency with `bundled` feature works for linking but creates two separate crate instances with different feature sets, causing type mismatch.
- **Fix:** Added `rusqlite` directly (without bundled initially, then with bundled for Windows), and wrapped the `call()` method to accept `FnOnce(&mut rusqlite::Connection) -> rusqlite::Result<T>` closures, internally mapping `rusqlite::Error` to `tokio_rusqlite::Error::Rusqlite(e)` before passing to the underlying `tokio_rusqlite::Connection::call()`.
- **Files modified:** spoon-backend/Cargo.toml, spoon-backend/src/control_plane/sqlite.rs
- **Verification:** cargo check and cargo test both pass
- **Committed in:** ea9509e (part of Task 1 commit)

## Issues Encountered
- The `tokio-rusqlite` crate defines its own `Error` enum wrapping `rusqlite::Error`, which creates a type-system boundary. Resolved by introducing a `call()` wrapper that bridges the two error types. This is a known ergonomic issue with `tokio-rusqlite` and the wrapper pattern is the standard solution.
