# Phase 02.1: SQLite Control Plane and Sync-Async Boundary - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 02.1 introduces SQLite as the backend control plane for Spoon runtime state and recovery metadata, and explicitly defines the sync-core / async-edge architecture boundary before deeper lifecycle splitting continues.
This phase exists to change the storage/control-plane direction in time for later lifecycle work. It is not a compatibility phase, and it is not a broad product feature phase.

</domain>

<decisions>
## Implementation Decisions

### SQLite Scope
- **D-01:** SQLite should be introduced as the backend control-plane database, not as a replacement for the filesystem as the runtime data plane.
- **D-02:** SQLite should directly own: canonical installed state, operation journal/checkpoints, doctor/repair metadata, bucket registry metadata, bucket sync metadata, in-flight operation/lock state, and schema/migration metadata.
- **D-03:** The filesystem should remain the source of truth for install roots, `current`, persist contents, cache/payloads, shims/shortcuts, and bucket repositories/manifests.

### Architecture Boundary
- **D-04:** `spoon-backend` should remain lifecycle-first and state-transition-first, not async-first.
- **D-05:** The backend should use a sync-core / async-edge split.
- **D-06:** Sync core means pure or near-pure business rules: state models, lifecycle planning, transition rules, ordering rules, validation, and read-model projection. It should not directly perform network, filesystem, process, or database-driver IO.
- **D-07:** Async edge means execution/adaptation layers: database facade, filesystem operations, process execution, network calls, and event emission.
- **D-08:** The preferred crossing point is “plan/decision first, then async execution” rather than each individual function deciding whether it is sync or async in isolation.

### SQLite Driver Choice
- **D-09:** Use `rusqlite` for the first SQLite integration and own the tokio async boundary inside backend store/repository modules.
- **D-10:** This choice preserves the SQLite/rusqlite control-plane model while avoiding dependence on an extra async bridge crate.
- **D-11:** Database access should still be fully encapsulated behind backend store/repository boundaries; lifecycle code should not directly depend on `rusqlite` details or manage blocking boundaries itself.

### Migration / Cutover Policy
- **D-12:** This phase should cut directly to SQLite as the control-plane state source; do not build a dual-read or dual-write compatibility layer for the old JSON state model.
- **D-13:** If old JSON state cannot be migrated or reconciled automatically, explicit operator repair is acceptable. Manual cleanup or deletion of old state is acceptable. Compatibility is not a goal.
- **D-14:** In short: direct cutover is preferred, and operator-visible repair is better than compatibility complexity.

### the agent's Discretion
- Planning may choose exact table layout, migration mechanics, and module names, as long as the storage split and sync/async boundary above remain intact.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `spoon-backend/src/scoop/state/` already defines the canonical installed-state model and projection boundaries.
- `RuntimeLayout` already separates derivable paths from persisted state, which is compatible with a SQLite control plane.
- `spoon-backend/src/scoop/doctor.rs` already surfaces stale/legacy state concerns and is a natural home for SQLite-backed repair metadata.

### Established Patterns
- The repo already treats backend state as a control concern and filesystem artifacts as runtime facts.
- The current architecture is moving toward thin app adapters and backend-owned orchestration.
- Future lifecycle work in Phase 3 should consume a better control-plane substrate instead of inheriting more JSON-file coordination logic.

### Integration Points
- `spoon-backend/src/scoop/state/*` is the most direct replacement target for the first SQLite-backed store.
- `spoon-backend/src/scoop/runtime/actions.rs` and later Phase 3 lifecycle modules will need journal/checkpoint integration.
- `spoon-backend/src/scoop/query.rs`, `info.rs`, and doctor surfaces will need to read from the SQLite control plane rather than JSON files once cutover happens.

</code_context>

<specifics>
## Specific Ideas

- The cleanest end state is filesystem-as-data-plane and SQLite-as-control-plane.
- `rusqlite` should be wrapped so tokio/blocking details stay inside backend store/repository edges rather than business logic.
- A direct cutover is preferred even if that means operator-visible repair or manual old-state deletion.

</specifics>
