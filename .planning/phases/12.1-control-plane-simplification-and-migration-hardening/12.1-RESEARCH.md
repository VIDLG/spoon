# Phase 12.1: Control Plane Simplification and Migration Hardening - Research

**Researched:** 2026-04-03
**Domain:** Simplifying control-plane boundaries and replacing handwritten migration plumbing
**Confidence:** HIGH

## Summary

This phase is best treated as infrastructure simplification with two goals:

1. make the control-plane API path-first and more honest
2. replace the bespoke migration runner with a standard SQL-migration mechanism

The current implementation already hints at the intended design:

- `ControlPlaneDb::open(db_path)` is the clean core entry
- SQL migrations are already versioned files
- most of the complexity comes from thin wrappers and duplicated path derivation

That makes this phase a strong fit for:

- moving DB-path derivation into layout
- deleting `open_for_layout(...)`
- simplifying `ControlPlaneDb`
- adopting `refinery` while keeping versioned SQL files

## Phase Requirements

| ID | Description | Planning Implication |
|----|-------------|----------------------|
| BECT-07 | Control-plane access is path-first at its core | Remove layout-first opening patterns from the core API. |
| BECT-08 | Control-plane migrations are simplified and hardened | Replace handwritten migration plumbing with a standard migration framework. |

## Recommended Plan Order

1. Move DB-path ownership into layout and simplify `ControlPlaneDb`
2. Update store/opening call sites away from `open_for_layout(...)`
3. Replace handwritten migration plumbing with `refinery`
4. Re-verify control-plane usage and record the phase

## Why `refinery`

`refinery` fits the current project shape well because:

- migrations are already SQL files
- the project uses `rusqlite`, so a migration-only crate is preferable to introducing a larger DB stack
- it reduces bespoke logic without changing the underlying storage model

## Current Code Reality

- `sqlite.rs` is doing too much of the layout-path story for what should be a path-first DB facade.
- `ControlPlaneLocation` currently behaves more like an unnecessary wrapper than a meaningful strategy type.
- `call_write(...)` is currently a semantic duplicate of `call(...)`.
- handwritten migration bookkeeping is still small enough to replace cleanly before it becomes entrenched.

## Validation Focus

- Control-plane opening is clearly path-first.
- Layout owns DB-path derivation.
- Migration handling is simpler and no longer depends on ad hoc runner logic.
- Existing Scoop/MSVC control-plane users still work after the cleanup.
