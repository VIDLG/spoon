# Phase 12.1: Control Plane Simplification and Migration Hardening - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning

<domain>
## Phase Boundary

This phase simplifies the backend control-plane layer so it is path-first at its core, removes thin layout-coupled convenience APIs from the core database boundary, and hardens the migration mechanism without turning the phase into a general database-platform rewrite.

This is infrastructure cleanup inside the current backend architecture, not a new product feature and not a repository-wide storage redesign.

</domain>

<decisions>
## Implementation Decisions

### Core API Direction
- **D-01:** The control-plane core should depend on a database path, not on `RuntimeLayout`.
- **D-02:** `ControlPlaneDb::open(db_path)` is the true core entry and should remain the primary constructor.
- **D-03:** `open_for_layout(...)` should be removed rather than preserved as a first-class API.

### Layout Ownership
- **D-04:** Database-path derivation belongs to layout, not to `control_plane`.
- **D-05:** `db_path_for_layout(...)` should move out of `control_plane/sqlite.rs` and into `RuntimeLayout` / `ScoopLayout` as a layout-owned helper such as `layout.scoop.control_plane_db_path()`.

### ControlPlaneDb Simplification
- **D-06:** `ControlPlaneDb` should be structurally simplified.
- **D-07:** `ControlPlaneLocation` is a likely over-abstraction and should be removed unless execution reveals a real distinction worth preserving.
- **D-08:** `call_write(...)` should not survive as a pure forwarding alias; either it gains real semantics or it is removed in favor of one primary connection-execution API.

### Migration Strategy
- **D-09:** Replace the ad hoc migration runner with `refinery`.
- **D-10:** Keep SQL migration files with explicit incremental versions (e.g. `0001_*.sql`, `0002_*.sql`) instead of switching to Rust-coded migrations.
- **D-11:** Upgrade the migration mechanism, not the migration-file philosophy.

### Naming
- **D-12:** The `control_plane` module name is good enough and should remain. The cleanup target is API shape and responsibility boundaries, not renaming the subsystem.

### External Libraries
- **D-13:** Using a mature migration crate is explicitly allowed here because it directly simplifies infrastructure behavior and future schema growth.
- **D-14:** Avoid introducing broader database abstractions beyond what is needed to improve the migration/control-plane boundary.

### the agent's Discretion
- The exact replacement shape for `call` / `call_write` can be decided during implementation, as long as the resulting API is simpler and more honest.
- The exact layout method name for the DB path can be chosen during implementation, as long as path ownership clearly belongs to layout.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ControlPlaneDb::open(db_path)` already exists and is the clean core constructor.
- Schema files are already SQL-based and versioned, which makes migration-framework adoption straightforward.

### Established Patterns
- The repo increasingly prefers path/layout truth to live in layout modules and for backend subsystems to consume those paths rather than deriving them themselves.
- Previous phases have already preferred deleting thin convenience wrappers over preserving weak abstractions.

### Integration Points
- `spoon-backend/src/control_plane/sqlite.rs` currently mixes core DB behavior with layout-derived path helpers.
- `spoon-backend/src/control_plane/migrations.rs` is the migration-mechanism hotspot.
- Store modules in Scoop/MSVC currently call `open_for_layout(...)` heavily, so this phase will need coordinated call-site cleanup.

</code_context>

<specifics>
## Specific Ideas

- Treat this as a path-boundary and migration-mechanism cleanup, not as a switch away from SQLite.
- Prefer simpler, more honest APIs even if it means more call sites explicitly derive the DB path from layout.
- Use `refinery` if it lets us delete bespoke migration plumbing rather than layering it on top.

</specifics>

<deferred>
## Deferred Ideas

- Broader data-model restructuring of SQLite tables remains outside this phase.
- Any deeper relationalization of Scoop JSON state fields stays deferred unless directly required later.

</deferred>
